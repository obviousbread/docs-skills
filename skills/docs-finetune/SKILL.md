# docs-finetune - обогащение примеров реальными документами

> **Gemini CLI:** use native Gemini file, shell, and web tools. Субагентов нет. `org_details` не требуется.
>
> **Codex:** use native Codex file and shell tools. Субагентов нет. `org_details` не требуется.

Читает папку с документами организации, извлекает структурные паттерны и сохраняет их в `~/.docs-plugin/{category}/examples/{category}.md`. Файл gitignored - содержит org-специфичные данные. Скилы ord/letter/memo автоматически используют его, если он существует.

## 0. Карта references

| Файл | Когда читать |
|------|-------------|
| `../docs-ord/references/examples.md` | При finetune ОРД - понять формат existing-примеров |
| `../docs-letter/references/letter-patterns.md` | При finetune писем - понять формат existing-примеров |
| `../docs-memo/references/examples.md` | При finetune служебок - понять формат existing-примеров |

## 1. Алгоритм

### Шаг 1. Уточнить параметры

Задай вопросы (можно одним сообщением):

1. **Категория:** ord (приказы, распоряжения, указания) / letter (письма) / memo (служебные записки)
2. **Папка с документами:** укажи путь к папке с исходными документами.
3. **Режим записи** (если `~/.docs-plugin/{category}/examples/{category}.md` уже существует):
   - **слияние** (по умолчанию) - новые элементы добавляются к существующим, дубли отфильтровываются
   - **перезапись** - файл заменяется полностью

Если `~/.docs-plugin/{category}/examples/{category}.md` не существует, режим записи не спрашивать - всегда создание.

### Шаг 2. Просканировать папку

Найди файлы с расширениями: `.docx`, `.txt`, `.md`.

Пропусти:
- `.doc` (старый бинарный формат, нет reader без LibreOffice)
- Временные файлы (`~$*.docx`)
- Системные файлы (`.DS_Store`)

### Шаг 3. Прочитать файлы

**Для `.docx`** - извлекай текст через python-docx:

```python
python3 -c "
from docx import Document
doc = Document('<путь>')
for p in doc.paragraphs:
    if p.text.strip():
        print(p.text)
for t in doc.tables:
    for row in t.rows:
        print(' | '.join(c.text.strip() for c in row.cells))
"
```

**Для `.txt` и `.md`** - Read tool напрямую.

### Шаг 4. Классифицировать

Для каждого файла определи, соответствует ли он выбранной категории:

| Категория | Признаки |
|-----------|----------|
| ОРД | Содержит «ПРИКАЗЫВАЮ:», «ОБЯЗЫВАЮ:», «ПРЕДЛАГАЮ:» или заголовок «ПРИКАЗ» / «РАСПОРЯЖЕНИЕ» / «УКАЗАНИЕ» |
| Письмо | Блок адресата + обращение «Уважаем*» или «Глубокоуважаем*» |
| Служебка | Содержит «СЛУЖЕБНАЯ ЗАПИСКА» или комбинацию «Кому:» + «От:» |

Если >20% файлов не соответствуют категории - покажи список несоответствующих и спроси: пропустить или включить.

### Шаг 5. Извлечь данные

Из matching-файлов извлекай данные, специфичные для категории. Правила ниже.

### Шаг 6. Показать превью

Покажи краткий отчет:
- Файлов прочитано / пропущено
- Количество извлеченных элементов по типам
- В режиме слияния: сколько новых / сколько отфильтровано как дубли
- 2-3 примера из каждой группы

Жди подтверждения пользователя перед записью.

### Шаг 7. Записать файл примеров

Путь: `~/.docs-plugin/{category}/examples/{category}.md`

Создай директорию если не существует: `mkdir -p ~/.docs-plugin/{category}/examples`

**Режим перезаписи:** заменить файл целиком новым содержимым.

**Режим слияния (по умолчанию):**
1. Прочитать существующий `~/.docs-plugin/{category}/examples/{category}.md` (если есть).
2. Для каждого нового элемента проверить: есть ли в существующем файле текст с похожестью ≥ 95% (сравнение по символьным n-граммам или простому ratio). Если да - пропустить.
3. Добавить прошедшие проверку элементы в конец соответствующих разделов.
4. Записать объединённый файл.

Алгоритм сравнения (Python, без внешних зависимостей):
```python
from difflib import SequenceMatcher

def is_duplicate(new_text, existing_texts, threshold=0.95):
    for existing in existing_texts:
        ratio = SequenceMatcher(None, new_text.strip(), existing.strip()).ratio()
        if ratio >= threshold:
            return True
    return False
```

### Шаг 8. Доложить

В режиме слияния: «Добавлено N новых элементов, пропущено M дублей. Файл: `~/.docs-plugin/{category}/examples/{category}.md`»
В режиме перезаписи: «Записано N элементов в `~/.docs-plugin/{category}/examples/{category}.md`. Скил использует их автоматически.»

## 2. Что извлекать по категориям

### ОРД

**Названия приказов** - полный заголовок после «О (Об)» (короткие, масштабируются без проблем).

**Основания (basis)** - только формулировки, НЕ содержащие ссылок на НПА (номеров, дат, названий нормативных актов). НПА пробивает npa-verifier, хранить их в примерах - засорять файл устаревающими данными. Сохранять формулировки типа: «В целях обеспечения...», «В связи с необходимостью...», «По результатам проверки...», «Для организации работы по...».

**Формулировки пунктов** - все, кроме типовых. Типовые НЕ сохранять:
- «Контроль за исполнением настоящего приказа возложить на...»
- «Контроль за исполнением оставляю за собой»
- «Секретарю опубликовать, разместить...»
- «Делопроизводителю ознакомить заинтересованных лиц...»
- «Приказ вступает в силу с момента подписания»
- «Настоящий приказ довести до сведения...»

Все остальные формулировки - сохранять.

**Приложения** - сохранять ВСЕ целиком в md-разметке. Приложения - самая трудоемкая часть ОРД; org-специфичные примеры наиболее ценны. В .docx приложения отделяются section breaks - каждый блок после break = отдельное приложение. Конвертировать в markdown: заголовки, таблицы, нумерованные списки, текст.

**Листы ознакомления** - НЕ сохранять (формируются автоматически по правилам скила, форма таблицы предопределена).

### Письма

Плоский список, без классификации по типам.

**Адресаты** - должность + ФИО + ученые степени (при наличии). Только уникальные записи (дедупликация по ФИО). Формирует «контактную книгу» из реальной переписки.

**Тела писем** - целиком (письма, как правило, короткие; полный контекст ценнее выдержек). Рядом с каждым телом указать подписанта от текущей организации (должность + ФИО).

### Служебки

**Только тело** - целиком, без блоков «Кому» и «От кого».

«Кому» предсказуем (руководитель организации), «от кого» - текущий автор. Суть примера - в формулировках тела документа.

## 3. Формат файлов примеров

### ОРД

```markdown
# Локальные примеры ОРД (gitignored - добавлены docs-finetune)

## Названия приказов

- О назначении ответственных за ...
- Об утверждении плана мероприятий по ...
- О создании комиссии по ...

## Основания (без НПА)

### L1: [краткое описание контекста]

[формулировка основания без ссылок на нормативку]

### L2: ...

## Формулировки пунктов

### Из приказа «[название]»

- [формулировка пункта]
- [формулировка пункта]

## Приложения

### Приложение из приказа «[название]»: [тип приложения]

[полный текст приложения в md-разметке: заголовки, таблицы, списки]
```

### Письма

```markdown
# Локальные примеры писем (gitignored - добавлены docs-finetune)

## Адресаты

| Должность | ФИО | Ученая степень |
|-----------|-----|----------------|
| Заместитель директора ФГБУ ... | [Фамилия И.О.] | д.м.н. |

## Письма

### Письмо 1
**Подписант:** [должность], [ФИО]

[полный текст тела письма]

### Письмо 2
...
```

### Служебки

```markdown
# Локальные примеры служебных записок (gitignored - добавлены docs-finetune)

### Служебка 1: [тема / первые слова]

[полный текст тела]

### Служебка 2: ...
```

## 4. Ограничения

- Файлы `.doc` (старый формат Word 97-2003) не поддерживаются - нужен LibreOffice для конвертации. Если в папке только `.doc`, предложи пользователю сконвертировать: `libreoffice --headless --convert-to docx *.doc`.
- По умолчанию при повторном запуске используется режим слияния с дедупликацией (порог 95%). Режим перезаписи - только по явному выбору пользователя.
- Зависимости: `python-docx` (должен быть доступен в Python окружении пользователя).
