# Чек-лист ревью качества ДИ

Запускается **всегда** после генерации `.docx`, до показа пользователю. Если хотя бы один пункт не пройден — исправить и перегенерировать (новый файл с суффиксом ` 2`, оригинал не трогать).

Чек-лист состоит из четырёх содержательных проверок и одной формальной. Содержательные проверки требуют чтения сгенерированного `.docx` (через python-docx), сверки с источниками (выбранный ПС, переданное Положение о СП) и `tier_rules.md`.

## 1. Покрытие ОТФ/ТФ из профстандарта

**Что проверяем:** все ТФ выбранного ПС (применимые к должности) отражены в разделах 2 «Трудовая функция» и 3 «Должностные обязанности».

**Как проверить:**
1. Загрузить список ТФ выбранного ПС (из шага «Поиск ПС», см. `profstandart-rules.md`).
2. Для каждой ТФ — найти соответствующий пункт в разделе 2 (формат «<Должность> осуществляет / организует / обеспечивает ...»).
3. Для каждой ТФ — найти соответствующие ТД в разделе 3.
4. Если какая-то ТФ не имеет отражения — вернуться к разделу 2/3 и добавить.

**Допустимые исключения:**
- ТФ, объективно не выполняемые на этой должности в Учреждении (например, ТФ «Проведение клинических исследований» в ПС, выбранном для немедицинской должности). Зафиксировать в отчёте: «ТФ X.YY опущена — не применима к должности в данном Учреждении».

**Не допускается:**
- Опущена ТФ без обоснования.
- В разделах 2/3 есть пункты, не имеющие источника в ПС или Положении о СП. Это не критично, но в отчёте отметить «дополнено сверх ПС из Положения».

## 2. Соответствие Положению о структурном подразделении

Проверяется, если пользователь передал Положение о СП.

**Что проверяем:** функции и задачи СП, описанные в Положении, отражены в ДИ через соответствующие пункты разделов 2 и 3 — с поправкой на роль (специалист выполняет действия, начальник отдела организует, начальник управления координирует).

**Как проверить:**
1. Извлечь из Положения о СП разделы «Цели и задачи», «Функции».
2. Для каждой функции СП:
   - если у должности есть ответственность за неё — найти пункт в разделе 2/3 ДИ;
   - если функция выходит за рамки роли (например, организационная функция СП в ДИ специалиста) — может быть опущена либо отражена через формулировку «принимает участие в».
3. Если Положение упоминает специфические зоны ответственности (например, профильные информационные системы, конкретные программы), а в ДИ их нет — добавить.

**Допустимые исключения:**
- Функции СП, явно делегированные другому отделу или другой должности внутри СП.

## 3. Корректность иерархии (1.4-1.5)

**Что проверяем:** пункты 1.4 (назначение) и 1.5 (подчинение) соответствуют тиру должности и фактической оргструктуре.

**Как проверить:**
1. Определить тир из названия должности (см. `tier_rules.md`).
2. Сверить шаблон 1.4 с матрицей в `tier_rules.md` (X — представление, Y — согласование).
3. Сверить шаблон 1.5 с матрицей в `tier_rules.md` (X — непосредственное, Y — при отсутствии).
4. Проверить названия вышестоящих СП и должностей (Управление, Отдел, замГД по направлению) на консистентность с тем, что указано в шапке ДИ.

**Не допускается:**
- Подчинение «на уровень ниже» нормы тира (например, нач. отдела подчиняется нач. отдела).
- Назначение по представлению лица, иерархически ниже целевой должности.
- Циклы в иерархии.

## 4. Объём прав по тиру (раздел 4)

**Что проверяем:** раздел 4 содержит права, соответствующие тиру, не больше и не меньше.

**Как проверить:** сверить с матрицей раздела 4 в `tier_rules.md`.

**Не допускается:**
- Специалист имеет право давать поручения подчинённым работникам — конфликт с тиром.
- Специалист инициирует проверки в подразделениях и филиалах — выход за рамки полномочий.
- Начальник отдела представляет всё Управление (если это не делегировано) — некорректно.
- Раздел 4 не содержит финального пункта «<Должность> пользуется всеми правами в соответствии с Трудовым кодексом Российской Федерации».

## 5. Формальные правила (по тексту, который агент кладёт в `sections`)

`generate.py` сам контролирует: шрифт, размер, межстрочный, выравнивание, отступы, цвет, нумерацию (арабская, без пропусков), маркер дашевых перечислений (`–`), гриф УТВЕРЖДАЮ, заголовок ДИ, таблицы подписей, лист ознакомления, авто-замену `—` → `–` и `ё` → `е`. Эти пункты в чек-лист НЕ выносятся — невозможно ошибиться, если использовать `create_di`.

Проверяет агент:

- [ ] Кавычки — только «ёлочки»; внутри — "лапки". Генератор не подменяет.
- [ ] Нет слешей между кириллическими словами как разделителя. Допустимо: `№ п/п`, номера документов. Генератор предупреждает, но не правит.
- [ ] Нет точки с запятой в обычных предложениях (только в перечнях).
- [ ] Каждый дашевый элемент в `dash_items` с маленькой буквы, заканчивается `;`; последний — `.`.
- [ ] Полное наименование должности и учреждения в `position_full` корректно (полное юр. название по ЕГРЮЛ).
- [ ] Состав цепочки `approvers` соответствует тиру (см. `tier_rules.md`) и реальной иерархии СП.

## Формат отчёта

После прохождения чек-листа выдать пользователю короткий отчёт:

```
Ревью качества ДИ <короткое наименование должности>:

[OK] 1. Покрытие ПС: 8 ТФ из 8 отражены.
[OK] 2. Положение: 5 функций СП покрыты (3 в разделе 2, 5 в разделе 3).
[OK] 3. Иерархия 1.4-1.5: соответствует тиру dept_head.
[OK] 4. Права: 8 пунктов, соответствуют тиру.
[OK] 5. Формальные правила: ошибок не найдено.

Использовано:
- Профстандарт: 07.003 «Специалист по управлению персоналом» (приказ Минтруда от 09.03.2022 № 109н)
- Положение о СП: <путь к файлу, если был передан>
- Тир: dept_head

Файл: <output_dir_di>/ДИ начальник отдела УП.docx
```

Если есть проблемы — `[FAIL]` или `[WARN]` с описанием. После исправления — повторить ревью.
