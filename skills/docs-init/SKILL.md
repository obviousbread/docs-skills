# docs-init — Инициализация docs

> **Gemini CLI:** use native Gemini file and shell tools. Read and write `~/.docs-plugin/org_details.md` explicitly.
>
> **Codex:** use native Codex file tools. Read and write `~/.docs-plugin/org_details.md` explicitly.

## Алгоритм

### 1. Проверить наличие org_details.md

Прочитай файл `~/.docs-plugin/org_details.md`.

**Если файл существует:** загрузи его содержимое в контекст и сообщи:
```
docs: реквизиты организации загружены.
```
На этом работа скилла завершается. Не задавай лишних вопросов.

**Если файл не существует:** переходи к шагу 2.

---

### 2. Первый запуск — настройка реквизитов

Сообщи пользователю:

> docs требует настройки реквизитов организации. Они сохранятся в `~/.docs-plugin/org_details.md` и не пропадут при обновлении. Пройдем по полям - это займет минуту.

Прочитай `~/.docs-plugin/runtime/references/org_details.md.example` — он содержит все необходимые поля. Если runtime еще не установлен, используй `references/org_details.md.example` из текущего репозитория.

Задай вопросы блоками (не по одному полю за раз):

**Блок 1 — Организация:**
- Полное наименование (как в ЕГРЮЛ, в кавычках-«ёлочках»)
- Краткое наименование (аббревиатура)
- Юридический адрес (с индексом)
- Телефон и факс
- Email, ОКПО, ОГРН, ИНН/КПП

**Блок 2 — Руководитель:**
- Должность
- ФИО в именительном падеже (напр.: И.О. Фамилия)
- ФИО в родительном падеже (напр.: И.О. Фамилии)

**Блок 3 — Автор документов (от чьего имени готовятся):**
- Должность
- ФИО кратко (Фамилия И.О.)
- ФИО полностью
- Телефон

**Блок 4 — Визирующие по умолчанию (для ОРД, можно пропустить):**
- До 3 визирующих: ФИО + должность каждого

**Блок 5 — Список сотрудников (можно пропустить):**
- Полный путь к сводному .xlsx файлу со списком сотрудников (напр. `~/Documents/Списки сотрудников/Организация/сводный.xlsx`). Используется для проверки ФИО и должностей при генерации ОРД и писем.

**Блок 6 — Пути вывода:**
Укажи папки для каждого типа документов:
- Папка для ОРД (приказы, распоряжения, указания)
- Папка для писем
- Папка для служебных записок
- Папка для должностных инструкций (ДИ)
- Папка для протоколов совещаний

Если пользователь явно отказывается указывать пути, предупреди:
> Без указания папок все документы будут сохраняться в домашнюю директорию (`~/`). Рекомендую указать папки сейчас — потом можно изменить через `docs-init`.

После предупреждения спроси ещё раз. Если пользователь снова отказывается — оставь поля пустыми и продолжи.

---

### 3. Записать org_details.md

После получения ответов сформируй файл `~/.docs-plugin/org_details.md` по структуре из `~/.docs-plugin/runtime/references/org_details.md.example`, подставив введенные значения вместо плейсхолдеров `{{...}}`. Если блок 5 пропущен - оставь `staff_file` пустым (строку не удалять). Если пользователь отказался от блока 6 - оставь поля `output_dir_*` пустыми (строки не удалять).

Используй инструмент Write для создания файла.

Сообщи пользователю:
```
docs: реквизиты сохранены в ~/.docs-plugin/org_details.md. Файл сохраняется при обновлении.
```

---

### 4. Обновление реквизитов и заметок

Если пользователь запросил обновление (а не первичную настройку), спроси, что нужно изменить или добавить.

**Изменение реквизитов** (`org_details.md`): прочитай текущий файл, покажи нужные значения, перезапиши файл целиком через Write с обновлёнными значениями. `org_details.md` содержит только структурированные поля `ключ: значение` — его безопасно перезаписывать.

**Добавление заметок об организации** (`org_notes.md`): если пользователь хочет добавить свободный контекст (структура, лицензии, функции, любые рабочие заметки) — добавь новый раздел `##` в конец `~/.docs-plugin/org_notes.md` через Edit. Если файл не существует — создай через Write с заголовком `# Заметки об организации`.
