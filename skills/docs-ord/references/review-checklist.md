# Чек-лист ревью ОРД — конформанс эталону generate.py

> **Это пол, а не потолок.** Чек-лист гарантирует, что механические инварианты формата не проскочат. Он НЕ ограничивает ревью. Главная петля ревью — холистическая: читай тот же корпус правил, что и при создании приказа (`npa-rules`, `basis-rules`, `doc-types`, `approval-rules`, `staff-rules`, `attachment-types`, `deadline-rules`, `amendment-rules`, `examples`, `org_details`, `~/DOCX Rules.md`), и ревьюй так, будто собираешь документ заново и сравниваешь с тем, что видишь. Выноси всё, что неверно, нелогично или можно сделать лучше — логику, обоснование, НПА, дубли, структуру, формулировки, стиль. Этот файл — лишь страховка снизу.

## Как проверять

Инварианты ниже задаёт `generate.py`. Сверка делается чтением XML документа (`document.xml`, `styles.xml`, `numbering.xml`), а не через `.text` python-docx — `.text` скрывает таблицы, разрывы, шрифты и нумерацию. Проверочный скрипт пишется на лету под эти инварианты (детерминированно, PASS/FAIL по каждому пункту).

При изменении `generate.py` этот файл синхронизируется (см. `maintenance.md`).

## A. Структурный слой (правится ТОЛЬКО пересборкой через generate.py)

Эти дефекты технически невозможно выразить через track-changes — они меняют схему документа. Ремонт = извлечь контент → `create_ord()` → эталонный `.docx`.

| Инвариант | Эталон (generate.py) |
|-----------|----------------------|
| Поля страницы | A4 21×29.7 см; поля top 2.0 / bottom 1.0 / left 3.0 / right 1.5 см |
| Шрифт по умолчанию | Times New Roman 14, `lang ru-RU`, интервал 1.0, отступы before/after 0 (docDefaults + стиль Normal) |
| Реквизиты (дата/№) | таблица 1×2 без границ, выравнивание таблицы по центру, столбцы 8 / 8.5 см (НЕ абзац с ручными пробелами) |
| Заголовок | таблица 1×2 без границ, cell margins 0 (НЕ абзацы) |
| Тип документа | «ПРИКАЗ/РАСПОРЯЖЕНИЕ/УКАЗАНИЕ» — жирный, по центру |
| Обоснование | justify, красная строка 1.25 см; ключевое слово (ПРИКАЗЫВАЮ:) — отдельным жирным абзацем, без запятой перед ним |
| Пункты | встроенная нумерация Word (`_list_para`), decimal, `rFonts` TNR + `sz 28` в каждом `lvl/rPr` |
| Подписант | таблица 1×2 без границ, cell margins 0, столбцы 8.5 / 8 см |
| Разрыв тело → согласование | разрыв РАЗДЕЛА (`sectPr type=nextPage`), НЕ разрыв страницы (`<w:br type=page>`) |
| Лист согласования | таблица 2 столбца без границ: «Проект <тип>а представил:», автор, «Проект <тип>а согласовали:», визирующие |
| Приложение | разрыв раздела + гриф (таблица 1×2 без границ) + заголовок по центру жирным; склонение «УТВЕРЖДЕН/О/А/Ы» по роду |
| Положение | многоуровневая нумерация: разделы «I.», «II.» (Roman, жирный, по центру), пункты «1.1.», подпункты «1.1.1.»; маркированные подсписки — «–» (bullet, en dash) |
| Таблица-перечень (form_table) | границы (`tblBorders single`), ширина 100% (`tblW pct 5000`, autofit), `cantSplit` на строках, шапка жирная по центру, шрифт ячеек TNR 14 |
| Лист ознакомления | отдельный раздел; столбцы РОВНО: «Наименование должности», «Подпись», «Расшифровка подписи», «Дата»; равная ширина 25%; TNR 14; ≥13 строк; сортировка по ФИО (А-Я) |

## B. Текстовый слой (правится track-changes в копии `<имя>_правки.docx`)

Автор ревизий = ключ `revision_author` из `org_details.md` (нет ключа — обезличенный дефолт).

| Инвариант | Правило |
|-----------|---------|
| Кавычки | только «ёлочки»; вложенные — "лапки"; прямые кавычки запрещены |
| Тире | дефис (-) или среднее (–); длинное (—) запрещено |
| Косая черта | не разделять слова через «/» (искл.: № п/п, ИНН/КПП, номера, коды) |
| Формат дат | ДД.ММ.ГГГГ; даты прописью («4 октября 2012») приводить к числовому виду |
| Перечисления | элементы через «;» внутри одного абзаца → маркированный список («–») |
| Ручные переносы | не использовать `\n`/Shift+Enter для подгонки строк в justify-абзацах |
| Ручные пробелы | не выравнивать пробелами/табами; убирать пробельные «дыры» |
| Точки/запятые | без двойной пунктуации («).,» → «),»), без середин­ных «;» вне списков |
| Орфография/грамматика | опечатки, согласование, «вследствие» vs «в следствие» и т.п. |
| Англицизмы | KPI → показатели результативности, дедлайн → срок исполнения и т.д. |
| Глаголы в пунктах | инфинитивы (Утвердить, Назначить, Обеспечить, …) |
| Формат НПА | кодексы / Конституция / ГОСТ / СанПиН — без «от … №…»; остальное — полные реквизиты; без заметок «(в ред. …)», «зарег. в Минюсте» |
| Алиас «(далее – X)» | если введён — употреблять последовательно; флагить смешение полного наименования и алиаса |

## Классификация находок

1. **Текстовый слой (B)** — собрать одним батч-предложением со списком конкретных правок. По «ОК» внести как track-changes в `<имя>_правки.docx` (оригинал не трогать).
2. **Структурный слой (A)** — описать в чате поэлементно, что изменится при пересборке, + обоснование. По «ОК» — пересобрать через `generate.py` в отдельный эталонный `.docx`.
3. **Содержательные находки холистической петли** (логика, дубли, висячие ссылки, контроль на исполнителей, слабое обоснование) — в чат, на обсуждение.

Не молчать ни о чём. Не плодить `.md`-отчёты — всё в чат.
